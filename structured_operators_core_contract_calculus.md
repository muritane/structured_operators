# Structured Operators: A Core Contract Calculus for Realizable Transitions

## Abstract

A practical action is not merely an edge between two abstract states.

It is an invocation of a maintained, typed, authorized, resource-bounded, and observable transition contract.

This document presents a compact calculus for such actions. Its central object is the **structured operator**: a stable public contract associated with one or more realizations that may be deployed, discovered, authorized, and executed in a concrete environment.

The calculus distinguishes:

```text
specification
realization
deployment
availability
executability
actual outcome
observed outcome
normative compliance
```

It provides judgments for:

```text
operator well-formedness
invocation typing
executability
sequential composition
parallel compatibility
failure-safe recovery
refinement
observational equivalence
plan validity
operator-constrained reachability
```

The core calculus is deliberately small. Authority, resources, time, references, knowledge, probability, concurrency, and institutional interpretation are modeled as explicit extensions rather than collapsed into an undifferentiated state relation.

The central principle is:

> Practical reachability is determined by the contracts an agent can correctly invoke, realize, observe, compose, and recover from under current constraints.

---

## 1. Motivation

A plain transition system contains states and relations:

\[
x \to x'
\]

This says that one state may follow another. It does not say:

```text
which operation denotes the transition
how the operation is invoked
which arguments occupy which roles
which preconditions must hold
which authority is required
which resources are consumed
which parts of state must remain unchanged
which failures may occur
which implementation performs the transition
which observations identify success
whether the invoker can determine what happened
```

In practical systems, transitions are generated by maintained schemas.

Examples include:

```text
machine instructions
API operations
financial transactions
laboratory procedures
transport services
administrative processes
measurements
human approvals
negotiation protocols
```

These are not arbitrary edges. They are constrained families of transitions with public interfaces, operational realizations, and context-dependent executability.

The appropriate unit of analysis is therefore not a state edge but a structured operator.

---

## 2. Design Goals

The calculus is intended to support five tasks.

### 2.1 Description

State precisely what an operator accepts, requires, changes, preserves, emits, consumes, and may fail to do.

### 2.2 Verification

Determine whether a realization satisfies an operator contract and preserves declared invariants.

### 2.3 Composition

Determine whether operator instances can be connected sequentially, conditionally, or concurrently.

### 2.4 Substitution

Determine whether one realization or operator safely replaces another under a declared observation regime.

### 2.5 Reachability

Determine whether a target condition can be achieved with sufficient reliability using the currently executable operator system.

The calculus is not intended to determine goals, values, legitimacy, or institutional desirability by itself.

---

## 3. Core Domains

Let the execution configuration be:

\[
c = (x,k,r,\alpha,e,t)
\]

where:

```text
x  = external or target-system state
k  = represented knowledge state
r  = resource state
α  = authority and capability state
e  = execution environment
t  = temporal context
```

The separation between \(x\) and \(k\) is fundamental.

An agent selects actions from its represented state \(k\), while operators execute against the actual state \(x\).

In general:

\[
k \not\equiv x
\]

The difference may arise through:

```text
stale observations
partial observability
incorrect models
ambiguous identifiers
concurrent change
measurement error
strategic deception
```

A practical semantics must therefore distinguish world transition from knowledge transition.

---

## 4. Structured Operators

A structured operator is a public contract together with a realization policy.

\[
u = (A_u,S_u,C_u,R_u,V_u)
\]

where:

```text
Au = access structure
Su = invocation signature
Cu = public contract
Ru = realization policy
Vu = version identity
```

### 4.1 Access structure

\[
A_u = (\operatorname{Id}_u,\operatorname{Namespace}_u,\operatorname{Locator}_u,\operatorname{Resolver}_u)
\]

Access determines how an operator is named, found, and resolved.

An identifier is not necessarily a locator. A name may require a resolver before a concrete realization can be reached.

### 4.2 Signature

\[
S_u = (\Sigma_u,\mathcal R_u,\mathcal T_u,\mathcal O_u)
\]

where:

```text
Σu = invocation grammar
Ru = argument roles
Tu = semantic argument types
Ou = declared result structure
```

### 4.3 Contract

\[
C_u = (P_u,Q_u,F_u,\Phi_u,I_u,E_u,O_u)
\]

where:

```text
Pu = admissibility predicate
Qu = success relation
Fu = failure relation
Φu = frame and effect specification
Iu = invariant obligations
Eu = effect summary
Ou = observation contract
```

### 4.4 Realization policy

\(R_u\) specifies how an invocation is associated with one or more maintained realizations.

It may define:

```text
fixed implementation selection
runtime service discovery
load balancing
jurisdictional routing
human assignment
fallback order
version negotiation
```

### 4.5 Version identity

A robust public identity is:

\[
V_u = (\text{name},\text{namespace},\text{version},\text{contract hash})
\]

This distinguishes label stability from semantic stability.

## 5. Invocation Language and Semantic Typing

Each operator defines a language of admissible invocation expressions.

\[
L(u)=\{a \mid a \text{ is accepted by } \Sigma_u\}
\]

An invocation must pass several layers:

```text
lexical validity
grammatical validity
role validity
semantic type validity
reference validity
contextual admissibility
```

The typing judgment is:

\[
\Gamma \vdash a : \operatorname{Invocation}(u)
\]

The environment \(\Gamma\) contains:

```text
variable bindings
semantic types
reference scopes
namespace assumptions
version assumptions
capability bindings
```

### 5.1 Roles are not representations

For:

```text
TRANSFER(source, destination, amount)
```

`source` and `destination` may share the same representation type while occupying different semantic roles.

Therefore:

\[
\operatorname{RepresentationEqual}(p,q)
\not\Rightarrow
\operatorname{RoleEqual}(p,q)
\]

### 5.2 Semantic and refinement types

A robust type system distinguishes:

\[
\operatorname{DistanceMeters}
\neq
\operatorname{DurationSeconds}
\]

although both may be encoded as 64-bit integers.

Refinement types may express local constraints:

\[
\operatorname{PositiveAmount}
=
\{a \in \operatorname{CurrencyAmount}\mid a>0\}
\]

State-dependent validity belongs to admissibility rather than static typing:

\[
a \leq \operatorname{AvailableBalance}(x)
\]

---

## 6. Admissibility and Executability

The contract contains a structured admissibility predicate:

\[
P_u(a,c)
=
P_u^{\text{state}}
\land
P_u^{\text{auth}}
\land
P_u^{\text{res}}
\land
P_u^{\text{time}}
\land
P_u^{\text{env}}
\]

The components are separated because they obey different propagation rules.

For example:

```text
state predicates are established by postconditions
authority predicates are established by delegation or capability effects
resource predicates are established by production, release, or reservation
time predicates are weakened by delay
environment predicates depend on deployment and infrastructure
```

The executability judgment is:

\[
c \vdash u(a)\;\operatorname{executable}
\]

A basic inference rule is:

\[
\frac{
\Gamma\vdash a:\operatorname{Invocation}(u)
\qquad
P_u(a,c)
\qquad
\exists m\;\operatorname{UsableRealization}(m,u,a,c)
}{
c\vdash u(a)\;\operatorname{executable}
}
\]

This separates public contract validity from practical realization availability.

---

## 7. Realizations

A realization is a maintained mechanism that purports to implement an operator.

Let:

\[
\operatorname{Implements}(m,u)
\]

mean that realization \(m\) satisfies the public contract of \(u\) under the declared verification regime.

Practical usability requires more:

\[
\operatorname{UsableRealization}(m,u,a,c)
\]

iff:

```text
Implements(m,u)
m is deployed in environment e
m is currently healthy enough
m is reachable from the invocation context
m supports the required operator version
m accepts the invocation instance a
required credentials and routes resolve
```

The following predicates should remain distinct:

\[
\operatorname{Implements}(m,u)
\]

\[
\operatorname{Deployed}(m,e,t)
\]

\[
\operatorname{Reachable}(m,a,c)
\]

\[
\operatorname{Healthy}(m,t)
\]

A realization can satisfy a contract while being undeployed, unreachable, unhealthy, or unavailable to a particular principal.

---

## 8. Lifecycle States

An operator may be:

### Specified

Its signature and contract are defined.

### Realized

At least one maintained realization satisfies the contract.

### Deployed

A satisfying realization is installed or assigned in an environment.

### Available

A deployed realization is reachable and resolvable in the present context.

### Executable

A particular invocation satisfies typing, admissibility, and realization conditions now.

Thus:

\[
\mathcal U_{\text{executable}}
\subseteq
\mathcal U_{\text{available}}
\subseteq
\mathcal U_{\text{deployed}}
\subseteq
\mathcal U_{\text{realized}}
\subseteq
\mathcal U_{\text{specified}}
\]

This chain prevents the false identification of specification with capability.

## 9. Operational and Normative Semantics

Practical systems often distinguish what happens from what is permitted or promised to happen.

### 9.1 Operational semantics

\[
\operatorname{OpSem}_m(a,c)
\]

describes what realization \(m\) may actually do.

### 9.2 Normative contract

\[
\operatorname{Norm}_u(a,c,c',z)
\]

describes what the public operator permits, requires, or guarantees.

### 9.3 Compliance

\[
\operatorname{Complies}(m,u)
\]

holds when the operational behavior of \(m\) refines the normative contract of \(u\) under the declared observation regime.

This distinction permits the calculus to represent:

```text
bugs
contract violations
unauthorized but physically effective actions
illegal institutional decisions
misconfigured deployments
fraudulent realizations
```

A transition can occur operationally without constituting a valid execution of the public operator.

---

## 10. Actual Outcome and Observed Outcome

Execution status and received observation are not identical.

A realization semantics is:

\[
\llbracket m:u(a)\rrbracket(c)
\in
\mathcal B(\mathcal C\times\mathcal S\times\mathcal O)
\]

where:

```text
C = resulting configuration
S = actual execution status
O = observation delivered to the invoker
B = behavior model
```

The actual status may be:

\[
s\in
\{\operatorname{Committed},
\operatorname{Aborted},
\operatorname{Partial},
\operatorname{Pending}\}
\]

The observation may be:

\[
o\in
\{\operatorname{Success}(y),
\operatorname{Failure}(f),
\operatorname{Timeout},
\operatorname{Unknown}\}
\]

The mapping between them is not necessarily one-to-one.

For example:

```text
actual status = Committed
observed response = Timeout
```

This distinction is required for correct reasoning about:

```text
retries
idempotency
reconciliation
exactly-once claims
unknown outcomes
distributed commit
human administrative delay
```

Knowledge state is updated by observation, not directly by hidden execution status:

\[
k' = \operatorname{UpdateKnowledge}(k,o)
\]

## 11. Behavior Models

Different forms of uncertainty require different semantics.

The calculus does not treat all variation as one generic distribution.

### 11.1 Deterministic behavior

\[
\mathcal B(X)=X
\]

### 11.2 Demonic nondeterminism

\[
\mathcal B(X)=\mathcal P(X)
\]

A guarantee must hold for every permitted result.

### 11.3 Angelic search

A plan succeeds if at least one permitted branch reaches the target.

This is useful for synthesis but not for adversarial execution guarantees.

### 11.4 Probabilistic behavior

\[
\mathcal B(X)=\operatorname{Dist}(X)
\]

Properties are measured by probability or expectation.

### 11.5 Epistemic uncertainty

The transition may be deterministic while the agent lacks sufficient knowledge to identify the result.

### 11.6 Strategic behavior

Another agent selects actions according to objectives, beliefs, and available strategies.

Strategic choice requires a game or mechanism model rather than ordinary nondeterminism.

A contract must declare which behavior interpretation applies.

---

## 12. Success, Failure, and Effects

A public operator defines separate relations for success and failure.

### 12.1 Success relation

\[
Q_u(a,c,c',y)
\]

### 12.2 Failure relation

\[
F_u(a,c,c_f,f,s)
\]

where \(s\) records whether the failure was aborted, partial, pending, or committed-but-unobserved.

### 12.3 Effect summary

\[
E_u=(\operatorname{Read}_u,\operatorname{Write}_u,\operatorname{Emit}_u,\operatorname{Consume}_u,\operatorname{Produce}_u)
\]

This summary supports static analysis, scheduling, and conflict detection.

### 12.4 Frame relation

\[
\Phi_u(c,c')
\]

states what must remain unchanged or within declared variation bounds.

An operator is defined partly by the state it is forbidden to modify.

---

## 13. Global Invariants

Let:

\[
I(c)
\]

be a global system invariant.

Examples include:

```text
conservation of funds
uniqueness of identifiers
referential integrity
capacity limits
separation of duties
physical safety constraints
legal consistency requirements
```

An operator is invariant-preserving when:

\[
I(c)
\land
P_u(a,c)
\land
Q_u(a,c,c',y)
\Rightarrow
I(c')
\]

Failure preservation may require:

\[
I(c)
\land
P_u(a,c)
\land
F_u(a,c,c_f,f,s)
\Rightarrow
I_f(c_f)
\]

where \(I_f\) is a weaker recovery invariant.

For non-atomic operators, a temporary invariant may be required during execution:

\[
I_{\text{transient}}(c_\tau)
\]

This permits intermediate states that are not publicly stable but remain recoverable.

## 14. Capability Semantics

Authority is modeled as typed capability state rather than a Boolean permission flag.

A capability may have the form:

\[
\kappa:
\operatorname{Capability}
\langle
\text{subject},
\text{operator},
\text{argument constraint},
\text{scope},
\text{expiry},
\text{delegation policy}
\rangle
\]

Capabilities may be:

```text
copyable
linear
consumable
revocable
delegable
attenuable
time-bound
context-bound
```

The authority admissibility predicate is:

\[
P_u^{\text{auth}}(a,c)
\iff
\alpha \models \operatorname{RequiredCapability}_u(a,c)
\]

An authority effect may:

```text
consume a one-use token
create a delegated capability
attenuate scope
revoke access
change ownership
open or close a role
```

### Principle of least transition

> Grant access to the narrowest operator contract and argument domain sufficient for the authorized objective.

Least privilege constrains who may act.

Least transition additionally constrains which state-change family their authority can denote.

---

## 15. Resource Semantics

Resources are typed quantities or capacities.

Examples include:

```text
money
energy
memory
inventory
bandwidth
time budget
attention
organizational capacity
legal quota
```

A resource predicate is:

\[
P_u^{\text{res}}(a,c)
\]

Resource effects are expressed as:

\[
r' = \operatorname{ResEff}_u(a,c,r)
\]

The effect may:

```text
consume
reserve
release
produce
transform
borrow
lock
```

Reservation and consumption are distinct.

A reserved resource may remain owned but unavailable to competing operators.

Composition must track both quantity and status.

---

## 16. Time Semantics

Time is not merely another scalar resource.

It affects validity, causality, deadlines, freshness, and availability.

An operator may declare:

```text
earliest start
latest start
deadline
maximum duration
expected duration
validity interval
cooldown
retry window
```

Temporal admissibility is:

\[
P_u^{\text{time}}(a,c)
\]

Sequential composition must prove that completion of one operator leaves the next operator's validity window open.

If \(T_u\) is a completion-time distribution and \(W_v\) is the valid start window of \(v\), then a probabilistic timing obligation may be:

\[
\Pr[T_u\in W_v] \geq \theta_t
\]

---

## 17. References and Resolution

Operators often produce handles rather than final values.

Examples include:

```text
user identifier
memory pointer
booking reference
legal entity number
transaction identifier
file descriptor
```

A handle is modeled by:

\[
h:
\operatorname{Handle}
\langle
\text{namespace},
\text{entity type},
\text{scope},
\text{version},
\text{authority class}
\rangle
\]

Resolution is itself an operator:

\[
\rho(h,c)
\to
\operatorname{Resolved}(\ell)
\mid
\operatorname{ResolutionFailure}(f)
\]

Reference continuity requires that a produced handle remain:

```text
in scope
unambiguous
resolvable
not revoked
semantically stable
authorized for the next use
```

An identifier that still resolves syntactically may nevertheless suffer semantic drift.

---

## 18. Information-Producing Operators

An information-producing operator primarily changes knowledge state.

Examples include:

```text
query
measurement
inspection
search
identifier resolution
legal interpretation request
inventory check
```

Its success relation may leave \(x\) unchanged while updating \(k\):

\[
x'=x
\]

\[
k'=k\cup\{o\}
\]

An observation contract should specify:

```text
accuracy
precision
freshness
coverage
confidence
provenance
resolution
known blind spots
failure modes
```

A freshness guarantee may be:

\[
t_{\text{now}}-t_{\text{observed}}\leq\delta
\]

Information quality affects executability because planning predicates are evaluated against represented knowledge.

## 19. Well-Formedness

The operator well-formedness judgment is:

\[
\vdash u\;\operatorname{wellformed}
\]

A core rule requires:

\[
\frac{
\operatorname{DecidableGrammar}(\Sigma_u)
\quad
\operatorname{RolesTyped}(S_u)
\quad
\operatorname{ResultsSeparated}(C_u)
\quad
\operatorname{EffectsCovered}(E_u,\Phi_u)
\quad
\operatorname{VersionStable}(V_u)
\quad
\operatorname{ObservationDefined}(O_u)
}{
\vdash u\;\operatorname{wellformed}
}
\]

Additional proof obligations may require:

```text
all referenced namespaces are declared
all capabilities have ownership and expiry semantics
all failure classes define post-failure state
frame conditions cover every relevant state component
resource effects are typed
invariant obligations are stated
realization verification criteria are declared
```

A malformed operator may have a usable implementation while remaining unsafe to compose.

---

## 20. Sequential Composition

Let \(u(a)\) execute before \(v(b)\).

The sequential compatibility judgment is:

\[
u(a)\triangleright v(b)
\]

The judgment requires proof obligations across several dimensions.

### 20.1 Semantic type continuity

Produced outputs and references must satisfy the semantic input requirements of \(v\).

\[
\operatorname{Type}(y_u)\preceq\operatorname{InputType}_v
\]

### 20.2 Postcondition-precondition implication

\[
Q_u(a,c,c',y)
\Rightarrow
P_v(b,c')
\]

When \(b\) depends on \(y\), the implication is interpreted under the binding induced by the plan.

### 20.3 Frame compatibility

The effects of \(u\) must preserve assumptions of \(v\) not explicitly re-established.

### 20.4 Invariant continuity

\[
I(c)\land Q_u\Rightarrow I(c')
\]

### 20.5 Resource continuity

The resource state after \(u\) must satisfy \(v\).

### 20.6 Authority continuity

Capabilities required by \(v\) must remain valid after \(u\).

### 20.7 Reference continuity

Handles passed to \(v\) must remain in scope and resolvable.

### 20.8 Temporal continuity

The completion of \(u\) must leave \(v\) temporally admissible.

### 20.9 Observation sufficiency

The observation emitted by \(u\) must provide enough knowledge to choose and bind \(v\) safely.

This is stronger than actual-state compatibility.

### 20.10 Failure compatibility

Every relevant failure or unknown outcome of \(u\) must lead to:

```text
a safe terminal state
an explicit recovery operator
an explicit reconciliation operator
or a declared unresolved state
```

A schematic rule is:

\[
\frac{
\vdash u\;\operatorname{wellformed}
\quad
\vdash v\;\operatorname{wellformed}
\quad
Q_u\Rightarrow P_v
\quad
\operatorname{FrameCompat}(u,v)
\quad
\operatorname{EffectCompat}(u,v)
\quad
\operatorname{FailureCompat}(u,v)
}{
u(a)\triangleright v(b)
}
\]

## 21. Sequential Hoare Rule

For deterministic total correctness, an operator instance may be summarized as:

\[
\{P\}\;u(a)\;\{Q\}
\]

Sequential composition follows:

\[
\frac{
\{P\}\;u(a)\;\{R\}
\qquad
\{R\}\;v(b)\;\{Q\}
}{
\{P\}\;u(a);v(b)\;\{Q\}
}
\]

For realistic operators, the intermediate assertion \(R\) must include:

```text
world state
knowledge state
resource state
capability state
reference scope
time bounds
failure markers
pending commitments
```

The ordinary Hoare rule remains valid, but the assertion language must be rich enough to express practical execution context.

---

## 22. Failure, Atomicity, and Unknown Outcomes

### 22.1 Atomic operator

An atomic operator guarantees:

```text
committed visible success
or
no committed external effect
```

Atomicity does not guarantee that the caller observes the correct status.

An atomic operation may commit while the caller receives a timeout.

### 22.2 Non-atomic operator

A non-atomic operator may expose partial effects.

Its contract should declare:

```text
commit points
partial-state classes
recovery invariant
retry semantics
compensation handles
irreversible effects
```

### 22.3 Idempotency

An operator is idempotent under observation regime \(\pi\) when:

\[
u(a);u(a)\equiv_{\pi}u(a)
\]

Idempotency may depend on an idempotency key or transaction identifier.

### 22.4 Reconciliation

When execution status is unknown, retry may be unsafe.

A reconciliation operator determines or narrows the actual state:

\[
\operatorname{RECONCILE}(h)
\to
\operatorname{Committed}
\mid
\operatorname{Aborted}
\mid
\operatorname{StillUnknown}
\]

Unknown-outcome recovery is therefore a first-class plan pattern.

---

## 23. Compensation

A compensating operator does not necessarily invert the original transition.

Let \(u\) be a forward operator and \(\bar u\) a compensation.

A compensation contract may guarantee:

\[
\operatorname{AcceptableRecovery}(c,\llbracket u;\bar u\rrbracket(c))
\]

rather than:

\[
\llbracket u;\bar u\rrbracket(c)=c
\]

Compensation may be lossy because:

```text
time has passed
fees were incurred
external observers saw the action
inventory changed
legal consequences attached
information was disclosed
```

A failure-safe workflow must state which effects are:

```text
reversible
compensable
reconcilable
irreversible
```

---

## 24. Parallel Composition and Interference

The parallel safety judgment is:

\[
u(a)\parallel v(b)\;\operatorname{safe}
\]

A basic conflict check uses read and write sets:

\[
\operatorname{Write}_u\cap
(\operatorname{Read}_v\cup\operatorname{Write}_v)
=
\varnothing
\]

and symmetrically for \(v\).

This condition is sufficient but not necessary.

Two operators may safely overlap when their effects commute or when a concurrency protocol preserves required invariants.

### 24.1 Rely-guarantee form

Each operator declares:

\[
\operatorname{Rely}_u
\]

and:

\[
\operatorname{Guarantee}_u
\]

Parallel composition requires:

\[
\operatorname{Guarantee}_u\Rightarrow\operatorname{Rely}_v
\]

\[
\operatorname{Guarantee}_v\Rightarrow\operatorname{Rely}_u
\]

### 24.2 Commutativity

A stronger condition is:

\[
u;v\equiv_{\pi,K}v;u
\]

Commutativity supports reordering, batching, and parallel execution.

### 24.3 Isolation declarations

A practical operator may declare:

```text
serializable
snapshot-isolated
read-committed
optimistic with conflict detection
exclusive-locking
best effort
```

The isolation level is part of the public contract when it affects observable correctness.

---

## 25. Refinement

Write:

\[
u'\preceq u
\]

when \(u'\) is a safe refinement of \(u\) under a declared guarantee set.

A standard refinement direction is:

### Preconditions

\[
P_u\Rightarrow P_{u'}
\]

The replacement accepts at least the situations accepted by the original.

### Success guarantees

\[
Q_{u'}\Rightarrow Q_u
\]

### Failure guarantees

\[
F_{u'}\Rightarrow F_u
\]

or every new failure is observationally irrelevant under the declared regime.

### Frame guarantees

\[
\Phi_{u'}\Rightarrow\Phi_u
\]

### Authority requirements

\[
\operatorname{AuthReq}_{u'}
\preceq
\operatorname{AuthReq}_u
\]

### Resource bounds

\[
\operatorname{Cost}_{u'}\leq\operatorname{Bound}_u
\]

### Observation compatibility

\[
O_{u'}\preceq O_u
\]

A faster or cheaper implementation is not automatically a refinement if it weakens reliability, privacy, precision, security, or failure behavior.

---

## 26. Observational Equivalence

Two operators are observationally equivalent relative to a projection \(\pi\) and guarantee set \(K\) when:

\[
u\equiv_{\pi,K}v
\]

The observation projection may include:

```text
functional outputs
state changes
timing
cost
failure classes
security events
privacy leakage
precision
resource consumption
audit records
```

Equivalence is relative because different users and systems observe different properties.

Two realizations may be equivalent for functional output while differing materially in:

```text
latency
jurisdiction
energy use
privacy
appeal rights
failure recovery
```

Operator identity is therefore contract-relative rather than absolute.

## 27. Plans

A plan is a guarded partial-order structure:

\[
P=(N,\prec,\gamma,\eta)
\]

where:

```text
N = operator instances
≺ = precedence relation
γ = guards and branch conditions
η = recovery, timeout, and compensation edges
```

A plan may contain:

```text
sequential steps
parallel branches
conditional execution
retries
timeouts
human approval gates
reconciliation
compensation
observation-dependent replanning
```

The plan-validity judgment is:

\[
\vdash P\;\operatorname{valid}
\]

A valid plan must prove:

```text
every invocation is well typed
every node is executable when reached
every sequential handoff is compatible
every parallel region is interference-safe
every branch guard is observable
every relevant failure has a declared continuation
all global invariants are preserved or recoverable
the target postcondition follows from terminal success states
```

---

## 28. Plan Execution

Plan execution is a relation over configurations, observations, and plan-control state.

\[
\operatorname{Exec}(P,c_0)
\]

may produce:

```text
successful terminal configurations
safe failed terminal configurations
pending configurations
unknown-outcome configurations
probability distributions over these classes
```

A plan is failure-safe when every reachable non-success state satisfies a declared recovery invariant:

\[
\forall c_f\in\operatorname{FailReach}(P,c_0),
\quad I_{\text{recovery}}(c_f)
\]

This does not require every failure to be automatically repaired.

It requires that failure states be bounded, observable enough, and governed by an explicit continuation policy.

---

## 29. Operator-Constrained Reachability

Let:

```text
c0 = starting configuration
U  = operator system
Y  = target condition
θ  = acceptable reliability threshold
```

The reachability judgment is:

\[
c_0,\mathcal U\vdash Y\;\operatorname{reachable}_{\theta}
\]

iff there exists a valid plan \(P\) such that:

\[
\Pr[\operatorname{Exec}(P,c_0)\models Y]\geq\theta
\]

subject to:

```text
operator realizations remain available
references resolve
capabilities remain valid
resource bounds hold
time constraints hold
failure states remain safe
required observations are obtainable
infrastructure assumptions remain satisfied
```

Reachability is therefore a property of:

```text
starting configuration
target condition
operator library
realization environment
knowledge state
authority state
resource state
time
acceptable risk
```

It is not a property of abstract states alone.

## 30. Optionality

Let:

\[
\mathcal U_{\operatorname{exec}}(c)
\]

be the executable operator instances from configuration \(c\).

Raw optionality may be approximated by:

\[
|\mathcal U_{\operatorname{exec}}(c)|
\]

but operator count is often misleading.

A better measure weights executable operators by future value, robustness, and target coverage:

\[
\operatorname{Optionality}(c)
=
\sum_{u(a)\in\mathcal U_{\operatorname{exec}}(c)}
 w(u,a,c)
\]

An alternative is goal-region coverage:

\[
\operatorname{Optionality}_G(c)
=
\mu\left(
\{Y\in G\mid c,\mathcal U\vdash Y\;\operatorname{reachable}_{\theta_Y}\}
\right)
\]

Buffers are valuable when they preserve a larger or more valuable future executable set.

Examples include:

```text
cash reserves
free memory
schedule slack
health
reputation
inventory buffers
legal continuity
transport capacity
```

A locally efficient action may be globally harmful when it collapses future executability.

## 31. Operator Promotion

A repeated plan may be packaged as a higher-level operator.

Let:

\[
P=u_n;\cdots;u_1
\]

A promoted operator \(U\) is justified when:

\[
\llbracket U\rrbracket
\equiv_{\pi,K}
\llbracket P\rrbracket
\]

and the boundary is:

```text
frequently reused
semantically coherent
stable enough to specify
valuable to expose
verifiable
recoverable
governable
maintainable
```

Promotion reduces:

```text
search burden
coordination burden
invocation complexity
reference handling
error rate
```

but may conceal:

```text
intermediate choices
failure points
resource consumption
jurisdictional routing
governance decisions
```

Promotion is therefore both an interface decision and a governance decision.

## 33. Mechanistic, Interpretive, and Strategic Operators

The calculus applies across domains, but outcome generation differs.

### 33.1 Mechanistic operators

Execution is governed primarily by computational or physical mechanism.

Examples:

```text
machine instructions
database transactions
cryptographic operations
controlled manufacturing steps
```

### 33.2 Interpretive operators

Authorized interpretation partly determines application of the contract.

Examples:

```text
medical assessment
legal adjudication
regulatory review
professional certification
```

The public contract may guarantee procedure, evidence standards, jurisdiction, review, and appeal rather than a predetermined outcome.

### 33.3 Strategic operators

Outcomes depend on interacting agents with independent objectives.

Examples:

```text
negotiation
auction
market transaction
collective decision
diplomatic process
```

These require explicit strategy, incentive, and information assumptions.

Strategic behavior should not be reduced to ordinary implementation nondeterminism.

## 34. Core Inference Judgments

A minimal calculus exposes the following judgments.

### Operator well-formedness

\[
\vdash u\;\operatorname{wellformed}
\]

### Invocation typing

\[
\Gamma\vdash a:\operatorname{Invocation}(u)

### Realization compliance

\[
m\models u
\]

### Executability

\[
c\vdash u(a)\;\operatorname{executable}
\]

### Sequential compatibility

\[
u(a)\triangleright v(b)
\]

### Parallel compatibility

\[
u(a)\parallel v(b)\;\operatorname{safe}
\]

### Recovery compatibility

\[
u(a)\dashrightarrow r(b)
\]

### Refinement

\[
u'\preceq u
\]

### Observational equivalence

\[
u\equiv_{\pi,K}v
\]

### Plan validity

\[
\vdash P\;\operatorname{valid}
\]

### Reachability

\[
c_0,\mathcal U\vdash Y\;\operatorname{reachable}_{\theta}
\]

These judgments make the framework reject invalid constructions rather than merely describe valid ones.

## 35. A Minimal Composition Checker

```text
CHECK-COMPOSITION(u, v, context):

    verify u and v are well formed

    verify bound outputs of u
        refine semantic inputs of v

    prove success postconditions of u
        imply state preconditions of v

    prove frame conditions of u
        preserve every assumption used by v

    prove global invariants survive u

    propagate resource effects of u
    verify remaining and reserved resources satisfy v

    propagate capability effects of u
    verify required authority for v remains valid

    verify produced handles:
        remain in scope
        resolve in the required namespace
        preserve semantic type and version
        retain sufficient authority

    propagate completion-time bounds
    verify v remains inside its validity window

    verify observations from u
        provide enough knowledge to bind and select v

    for every failure or unknown outcome of u:
        verify safe termination
        or identify reconciliation, recovery, or compensation

    verify permitted concurrent activity
        cannot invalidate the handoff assumptions

    return compatible
        with residual proof obligations
```

The checker may return conditional compatibility when obligations depend on runtime checks.

## 36. Worked Example: Payment Followed by Shipment

Consider:

```text
AUTHORIZE_PAYMENT(order, amount)
CAPTURE_PAYMENT(authorization)
CREATE_SHIPMENT(order)
```

### 36.1 Authorization operator

The authorization operator may guarantee:

```text
an authorization handle is produced
a spending amount is reserved
the handle expires after a time window
no funds are yet transferred
```

### 36.2 Capture operator

Capture requires:

```text
a valid authorization handle
unexpired reservation
matching order identity
sufficient merchant capability
```

It may return a timeout even if capture committed.

### 36.3 Shipment operator

Shipment creation may require confirmed payment knowledge, not merely actual committed payment.

Therefore:

```text
actual payment success
```

is insufficient when the observation is unknown.

A safe plan is:

```text
CAPTURE_PAYMENT
    success -> CREATE_SHIPMENT
    definite failure -> RELEASE_RESERVATION
    timeout/unknown -> RECONCILE_PAYMENT
        committed -> CREATE_SHIPMENT
        aborted -> RELEASE_RESERVATION
        unknown -> manual review
```

This example shows why composition depends on observed outcome, reference continuity, time, compensation, and recovery.

## 37. Worked Example: Institutional Decision

Consider:

```text
SUBMIT_APPLICATION(applicant, evidence, jurisdiction)
REVIEW_APPLICATION(case)
ISSUE_DECISION(case)
APPEAL_DECISION(decision)
```

The review operator is interpretive.

Its contract may guarantee:

```text
qualified decision-maker
admissible evidence process
conflict-of-interest rules
reasoned decision record
decision deadline
appeal eligibility
```

It may not guarantee approval or rejection in advance.

Operational and normative semantics must remain distinct because:

```text
a decision may be recorded but procedurally invalid
a reviewer may exceed jurisdiction
a deadline may be violated
a valid appeal right may remain despite operational finality
```

The operator contract therefore includes procedural obligations, authority provenance, audit records, and recovery through appeal.

## 40. Quality of an Operator System

Counting operators is insufficient.

A useful operator system should be evaluated by:

### Coverage

Which valuable target conditions become reachable?

### Composability

How often do contracts connect without custom adaptation?

### Safety

How tightly are authority, effects, and failures bounded?

### Stability

How resistant are identifiers and contracts to semantic drift?

### Discoverability

Can agents find the correct operator and version?

### Verifiability

Can realizations be checked against contracts?

### Recoverability

Are failures observable, bounded, reconcilable, or compensable?

### Maintainability

What infrastructure and institutional support are required?

### Cognitive cost

How difficult is correct operator selection and composition?

A rough leverage measure is:

\[
\operatorname{Leverage}(u)
=
\frac{
\text{valuable reliable compositions enabled by }u
}{
\text{maintenance cost}
+
\text{cognitive cost}
+
\text{security cost}
}
\]

Invalid, redundant, dangerous, and uninterpretable compositions should not be counted as useful capability.

## 41. Design Principles

### Explicit invocation

Every operator should expose a decidable invocation language.

### Semantic typing

Types should encode meaning, role, unit, scope, and version where relevant.

### Structured admissibility

State, authority, resources, time, and environment should be distinguishable because they propagate differently.

### Explicit frames

Contracts should define what may not change.

### Invariant preservation

Operators should state which global and recovery invariants they preserve.

### Actual-observed separation

Execution status and caller observation should not be conflated.

### Failure completeness

Every relevant failure, partial state, pending state, and unknown outcome should have declared semantics.

### Least transition

Expose the narrowest sufficient transition family.

### Capability continuity

Composition should track authority as carefully as data.

### Reference continuity

Handles should preserve scope, resolution, version, and meaning.

### Observable identity

Replacement should be justified relative to declared guarantees and observations.

### Behavior clarity

Deterministic, demonic, probabilistic, epistemic, and strategic uncertainty should remain distinct.

### Semantic maintenance

Documentation, versioning, deployment, verification, and governance are part of operational capability.

### Optionality preservation

Plans should account for their effect on the future executable operator set.

### Promotion with caution

Repeated paths should become primitives only when their boundaries are stable, verifiable, and governable.

## 42. Core Propositions

### Proposition 1

A practical operator is not an arbitrary state relation. It is a typed invocation family with a public contract, realization policy, observation semantics, and maintained implementation evidence.

### Proposition 2

Specification, realization, deployment, availability, and executability are distinct statuses.

### Proposition 3

Operational behavior and normative contract are distinct; noncompliant transitions remain representable.

### Proposition 4

Actual execution status and observed response are distinct, especially in distributed, delayed, and institutional systems.

### Proposition 5

Syntax and representation types are insufficient for composition; semantic roles, references, and contract implications are required.

### Proposition 6

Frame conditions and invariant obligations are essential because useful operators are defined partly by what they are forbidden to change.

### Proposition 7

Authority, resources, references, time, environment, and knowledge are components of execution configuration rather than external annotations.

### Proposition 8

Sequential composition is a proof obligation across state, knowledge, effects, resources, authority, references, time, observations, failures, and interference.

### Proposition 9

Parallel composition requires compatible interference assumptions, isolation guarantees, or commutative effects.

### Proposition 10

Failure behavior, pending state, unknown outcomes, retry semantics, and compensation are part of operator identity.

### Proposition 11

Refinement is relative to a declared observation and guarantee regime.

### Proposition 12

Resolvers, parsers, sensors, gateways, adapters, and reconciliation procedures are themselves operators.

### Proposition 13

Information-producing operators transform knowledge state and must declare freshness, accuracy, provenance, and uncertainty.

### Proposition 14

Mechanistic, interpretive, and strategic operators share a contract boundary but require different outcome-generation models.

### Proposition 15

Optionality is better understood as weighted future executability or reachable-goal coverage than as raw operator count.

### Proposition 16

Practical reachability is a property of a target, configuration, operator system, realization environment, and acceptable reliability threshold.

### Proposition 17

A rich operator calculus does not determine value, legitimacy, governance, or which operators should exist.

## 45. Compact Definition

A structured operator is:

\[
u=(A,S,C,R,V)
\]

It is **specified** when its signature and contract are defined.

It is **realized** when at least one maintained mechanism complies with that contract.

It is **deployed** when a compliant realization is installed or assigned in an environment.

It is **available** when a deployed realization is reachable and resolvable.

An invocation is **executable** when:

```text
its syntax is valid
its arguments satisfy semantic roles and types
its references resolve
its state preconditions hold
its capability requirements are satisfied
its resource obligations are feasible
its time constraints hold
a usable realization is available
```

Two operator instances compose when:

```text
the first establishes the second's semantic inputs
the first preserves the second's assumptions
global invariants remain valid
resources and capabilities remain sufficient
references remain meaningful
time windows remain feasible
observations are sufficient for the handoff
failures and unknown outcomes have safe continuations
concurrent interference is controlled
```

A target is practically reachable when a valid plan of such instances achieves it with acceptable reliability.

## 46. Final Principle

An operator name is not yet an operator.

It needs a stable signature and contract.

A contract is not yet a capability.

It needs a compliant realization.

A compliant realization is not yet usable.

It must be deployed, healthy, reachable, resolvable, authorized, and maintained.

A well-typed invocation is not yet executable.

Its state, authority, resource, time, environment, and reference conditions must hold.

An execution response is not necessarily the execution truth.

Actual status and observed status may diverge.

A successful local transition is not yet a safe trajectory.

Its effects, frames, invariants, references, observations, failures, and interference must compose with what follows.

A repeated trajectory is not yet infrastructure.

It must be stabilized, specified, verified, governed, deployed, and maintained as a reusable operator.

A finite agent does not act through arbitrary state change.

It acts through the structured transition contracts that remain executable in its present world.
