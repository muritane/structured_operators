# Structured Operators: A Contract Calculus for Composable Transitions

## 1. Purpose

Finite systems can support vast spaces of behavior without enumerating every possible state transition.

A processor does not store every arithmetic result.

An API does not list every possible database mutation.

A laboratory does not maintain every possible molecule.

A transport network does not contain a direct route between every pair of locations.

An institution does not separately encode every possible legal or administrative outcome.

Instead, these systems expose reusable transition schemas whose admissible instances are generated from compact rules.

The central object is therefore not merely a state, an edge, an action label, or a procedure name.

It is a **structured operator**:

```text
stable identity
+
typed invocation language
+
role-bearing arguments
+
execution context
+
preconditions
+
success and failure contracts
+
frame conditions
+
resource and authority effects
+
observable semantics
+
maintained realization
->
reusable family of constrained transitions
```

The central problem is:

```text
Which operator is intended?
Which invocations are well formed?
When is an invocation executable?
What transition does it promise?
What may it modify?
How can it fail?
When can it compose with another operator?
When may one implementation replace another?
Which targets are reachable through the available operator system?
```

This document develops a compact formal framework for answering those questions.

---

## 2. Why State Graphs Are Not Enough

A plain transition graph contains states and edges:

\[
x \to x'
\]

This representation is useful but incomplete.

It does not by itself explain:

```text
how an edge is named
how an agent invokes it
which arguments occupy which roles
which states admit execution
which authority is required
which resources are consumed
which parts of state must remain unchanged
which failures are permitted
which observations define equivalence
which implementation realizes the edge
```

In practical systems, an edge is rarely an unconstrained possibility.

It is normally generated by a maintained schema.

Therefore:

> Practical reachability is governed by structured transition contracts, not by abstract state adjacency alone.

## 3. The Four-Layer Operator Model

A practical operator is represented as:

\[
u =
(
\operatorname{Access}_u,
\operatorname{Signature}_u,
\operatorname{Contract}_u,
\operatorname{Realization}_u
)
\]

### Access

Access determines how an operator is identified, reached, resolved, and authorized.

\[
\operatorname{Access}_u
=
(
\operatorname{Id}_u,
\operatorname{Locator}_u,
\operatorname{Resolver}_u,
\operatorname{Capability}_u
)
\]

### Signature

The signature determines which invocation expressions are admissible.

\[
\operatorname{Signature}_u
=
(
\Sigma_u,
\mathcal R_u,
\mathcal T_u,
\mathcal O_u
)
\]

where:

```text
Σu
=
invocation grammar
```

```text
Ru
=
argument roles
```

```text
Tu
=
argument and result types
```

```text
Ou
=
declared output structure
```

### Contract

The contract determines what an admissible invocation means.

\[
\operatorname{Contract}_u
=
(
P_u,
Q_u,
\operatorname{Frame}_u,
\operatorname{Res}_u,
\operatorname{Auth}_u,
\operatorname{Time}_u,
\operatorname{Fail}_u,
\operatorname{Obs}_u
)
\]

### Realization

The realization is the maintained mechanism that executes the contract.

\[
\operatorname{Realization}_u
=
M_u
\]

The public identity of an operator normally depends most strongly on its access structure, signature, and contract.

The realization may change while the public operator remains stable.

## 4. A Canonical Semantic Model

The framework uses one general semantic form.

Let:

\[
a \in L(u)
\]

be a well-formed invocation of operator \(u\).

Let the execution configuration be:

\[
c =
(
x,
k,
r,
\alpha,
e,
t
)
\]

where:

```text
x
=
world or target-system state
```

```text
k
=
agent knowledge or represented state
```

```text
r
=
resource state
```

```text
α
=
authority and capability state
```

```text
e
=
external environment
```

```text
t
=
time and temporal context
```

The operator semantics are:

\[
\llbracket u(a) \rrbracket
:
\mathcal C
\to
\mathcal D
\left(
\operatorname{Result}(\mathcal C,\mathcal Y,\mathcal F)
\right)
\]

where:

```text
C
=
configuration space
```

```text
D
=
deterministic value, nondeterministic set, or probability distribution
```

```text
Y
=
observable success outputs
```

```text
F
=
observable failure values
```

A result is either:

\[
\operatorname{Success}(c',y)
\]

or:

\[
\operatorname{Failure}(c_f,f)
\]

This model covers:

```text
deterministic computation
probabilistic processes
network operations
institutional procedures
measurements
human-mediated decisions
physical transformations
```

It also makes failure an explicit semantic outcome rather than an external accident.

## 5. Invocation Languages

Each operator defines a local language of admissible invocations.

Let:

\[
\Sigma_u
\]

be the invocation grammar of \(u\).

Then:

\[
L(u)
=
\{
a
\mid
a \text{ is accepted by } \Sigma_u
\}
\]

For a register addition instruction:

```text
ADD <destination-register>, <source-register>, <source-register>
```

Examples in the language include:

```text
ADD r3, r1, r2
ADD r7, r7, r0
```

Examples outside the language include:

```text
ADD banana, r1, r2
ADD r1, r2
ADD r9000, r1, r2
```

The last example may be grammatically similar but type-invalid for an architecture with registers \(r_0\) through \(r_{31}\).

Validation therefore proceeds through layers:

```text
lexical validity
->
grammatical validity
->
role validity
->
type validity
->
context validity
->
precondition validity
->
authority validity
->
resource validity
->
execution
```

A valid sentence is not necessarily an executable invocation.

## 6. Roles, Types, and Semantic Refinement

Arguments occupy role-bearing positions.

For:

```text
TRANSFER(source, destination, amount)
```

the first and second arguments may share an account-reference representation while having different semantic roles.

Therefore:

```text
shared representation
does not imply
shared role
```

Likewise:

```text
64-bit integer
```

may encode:

```text
meters
seconds
euros
memory addresses
user identifiers
```

A robust type system distinguishes representation types from semantic types:

\[
\operatorname{DistanceMeters}
\neq
\operatorname{DurationSeconds}
\]

even when both are represented by the same bit pattern.

The framework permits refinement types:

\[
\operatorname{PositiveAmount}
=
\{
a \in \operatorname{CurrencyAmount}
\mid
a > 0
\}
\]

and context-dependent admissibility:

\[
a
\leq
\operatorname{AvailableBalance}(x)
\]

Parameters are therefore not free variables.

They are constrained by:

```text
syntax
role
type
semantic refinement
state
authority
time
resource availability
```

## 7. Preconditions, Postconditions, and Frame Conditions

A contract is not merely a mapping from input to output.

It specifies:

### Preconditions

What must hold before execution.

\[
P_u(a,c)
\]

### Success postconditions

What must hold after successful execution.

\[
Q_u(a,c,c',y)
\]

### Failure conditions

Which failures may occur and what state is permitted afterward.

\[
F_u(a,c,c_f,f)
\]

### Frame conditions

Which parts of configuration must remain unchanged.

\[
\operatorname{Frame}_u(c,c')
\]

For example, an addition instruction may specify:

```text
destination register changes
specified status flags may change
program counter advances
all unrelated registers remain unchanged
memory remains unchanged
privilege state remains unchanged
```

Without frame conditions, an operator could satisfy its advertised output while arbitrarily corrupting unrelated state.

Therefore:

> An operator is defined partly by what it is forbidden to change.

## 8. Resources, Authority, and Time Are First-Class

Practical operators transform more than target state.

They may consume or reserve:

```text
time
money
energy
memory
bandwidth
inventory
attention
legal quota
organizational capacity
```

They may also require or modify authority:

```text
access token
privilege level
license
ticket
delegation
role membership
ownership credential
```

Execution feasibility is therefore a predicate over the full configuration:

\[
\operatorname{Executable}(u,a,c)
\]

not merely over target state \(x\).

A typical definition is:

\[
\operatorname{Executable}(u,a,c)
\iff
\begin{cases}
a \in L(u) \\
\operatorname{Typed}_u(a) \\
P_u(a,c) \\
\alpha \models \operatorname{Auth}_u(a,c) \\
r \models \operatorname{Res}_u(a,c) \\
t \models \operatorname{Time}_u(a,c) \\
M_u \text{ is currently available}
\end{cases}
\]

Knowing an operator's name, location, or syntax is insufficient when the required capability, resource, or time window is absent.

## 9. Operator Lifecycle States

A useful framework distinguishes four operator statuses.

### Specified

A signature and contract exist.

```text
the operator has a defined meaning
```

### Realized

At least one implementation satisfies the contract.

```text
the operator can exist in practice
```

### Available

A satisfying realization is currently reachable in context.

```text
the operator can presently be accessed
```

### Executable

A particular invocation currently satisfies all conditions.

```text
the operator can be run now with these arguments
```

This gives the inclusion chain:

\[
\mathcal U_{\text{executable}}
\subseteq
\mathcal U_{\text{available}}
\subseteq
\mathcal U_{\text{realized}}
\subseteq
\mathcal U_{\text{specified}}
\]

The distinction prevents a common error:

```text
having a specification
!=
having a capability
```

## 10. A Well-Formedness Judgment

An operator definition should satisfy a structural judgment:

\[
\vdash u \; \operatorname{wellformed}
\]

Possible proof obligations include:

```text
the identifier is unambiguous within its namespace
the invocation grammar is decidable
every argument role has a declared type
success and failure results are distinguishable
frame conditions cover all relevant state
resource effects are defined
authority requirements are explicit
version identity is stable
observable behavior is defined
at least one realization can be verified against the contract
```

A malformed operator may have:

```text
ambiguous argument roles
undefined failure state
unbounded side effects
unstated authority requirements
unstable interpretation
unresolvable references
```

Well-formedness is therefore a prerequisite for safe composition.

## 11. Composition as a Proof Obligation

Suppose operator instance \(u(a)\) is followed by \(v(b)\).

Simple type matching is not enough.

The composition judgment is:

\[
u(a)
\triangleright
v(b)
\]

meaning:

```text
successful execution of u establishes
a valid execution context for v
```

This judgment requires several obligations.

### Type continuity

Outputs and produced references satisfy the semantic types expected by \(v\).

\[
\operatorname{Type}(y_u)
\preceq
\operatorname{InputType}_v
\]

### Postcondition-precondition implication

\[
Q_u
\Rightarrow
P_v
\]

### Frame compatibility

The effects of \(u\) preserve assumptions on which \(v\) depends.

### Resource feasibility

\[
r_{\text{after }u}
\models
\operatorname{Res}_v
\]

### Authority continuity

\[
\alpha_{\text{after }u}
\models
\operatorname{Auth}_v
\]

### Reference continuity

Handles produced by \(u\) remain:

```text
in scope
resolvable
authorized
semantically valid
```

### Temporal compatibility

The completion time of \(u\) leaves \(v\)'s validity interval open.

### Failure compatibility

Every relevant failure of \(u\) either:

```text
preserves a safe terminal state
or
establishes the preconditions of a recovery operator
```

### Interference compatibility

Concurrent activity cannot invalidate the assumptions of \(v\) between the two steps.

Thus:

\[
\operatorname{Compose}(u,v)
=
\text{types}
+
\text{semantics}
+
\text{frames}
+
\text{resources}
+
\text{authority}
+
\text{references}
+
\text{time}
+
\text{failures}
+
\text{interference}
\]

---

## 12. A Contract-Matching Algorithm

A composition checker can be expressed schematically.

```text
CHECK-COMPOSITION(u, v, context):

    verify u and v are well formed

    verify declared outputs of u
        refine the required inputs of v

    prove success postconditions of u
        imply preconditions of v

    prove frame conditions of u
        preserve every state predicate relied upon by v

    propagate resource effects of u
    verify remaining resources satisfy v

    propagate authority effects of u
    verify remaining capabilities satisfy v

    verify produced handles:
        remain in scope
        resolve in the target namespace
        retain required authority
        preserve semantic identity

    verify temporal windows overlap

    for each declared failure of u:
        verify safe termination
        or identify a valid recovery path

    verify no permitted concurrent operator
        can invalidate the handoff assumptions

    return compatible
        with generated proof obligations
```

This turns the framework from a descriptive vocabulary into a design and verification method.

## 13. Sequential, Parallel, and Conditional Plans

A plan is not always a linear chain.

Represent a plan as:

\[
P =
(
N,
\prec,
\gamma
)
\]

where:

```text
N
=
operator instances
```

```text
≺
=
precedence relation
```

```text
γ
=
guards, branches, and recovery conditions
```

If:

\[
u_i \prec u_j
\]

then \(u_i\) must complete before \(u_j\) begins.

Operators without an ordering relation may execute concurrently when their contracts are compatible.

A practical plan may contain:

```text
parallel branches
conditional choices
retries
timeouts
compensation
human approval gates
observation-dependent replanning
```

Trajectory design is therefore closer to a typed workflow or partial-order program than to a simple path.

## 14. Concurrency and Interference

Two operators may be individually executable and still be unsafe together.

Possible conflicts include:

```text
lost update
double spending
capacity oversubscription
race condition
deadlock
stale observation
revoked authority
reference invalidation
```

Concurrency contracts should specify:

```text
read set
write set
locking behavior
isolation level
serialization constraints
commutativity
retry semantics
conflict detection
```

A useful rely-guarantee form is:

\[
\operatorname{Rely}_u
\]

for assumptions about concurrent behavior, and:

\[
\operatorname{Guarantee}_u
\]

for effects \(u\) promises to respect.

Parallel composition requires:

\[
\operatorname{Guarantee}_u
\Rightarrow
\operatorname{Rely}_v
\]

and:

\[
\operatorname{Guarantee}_v
\Rightarrow
\operatorname{Rely}_u
\]

over the shared configuration.

Commutativity provides a stronger condition:

\[
u \circ v
\equiv
v \circ u
\]

When it holds, reordering and parallel execution become easier.

## 15. Failure, Atomicity, and Compensation

Failure behavior is part of operator identity.

Important questions include:

```text
Was state partially modified?
Which observations were emitted?
Were resources consumed?
Did authority change?
Can the invocation be retried safely?
Is the invocation idempotent?
Is compensation possible?
```

### Atomic operators

An atomic operator guarantees:

```text
complete visible success
or
no visible transition
```

### Non-atomic operators

A non-atomic operator may leave partial state.

Such an operator should declare:

```text
commit points
partial effects
recovery invariants
compensation handles
```

A workflow may therefore contain:

```text
forward operators
recovery operators
compensating operators
```

For example:

```text
reserve inventory
->
charge payment
->
create shipment
```

If shipment creation fails, compensation may require:

```text
refund payment
release inventory
```

A composition is safe only when failure trajectories are designed, not merely successful trajectories.

## 16. Information-Producing Operators

Not every operator primarily changes external state.

Some operators produce information:

```text
read register
query database
measure temperature
resolve identifier
inspect inventory
search route
request legal interpretation
```

Such an operator may update knowledge state:

\[
k'
=
k
\cup
\{o\}
\]

while leaving target state mostly unchanged.

Execution still occurs against the actual world state \(x\), while selection is often based on represented state \(\hat{x}\) contained in \(k\).

This creates a model gap:

\[
\hat{x}
\neq
x
\]

Possible consequences include:

```text
stale references
invalid precondition assumptions
incorrect route selection
misidentified resources
unexpected failure
```

Therefore information quality is part of practical executability.

Observation contracts should specify:

```text
accuracy
precision
freshness
coverage
confidence
provenance
failure modes
```

## 17. References and Resolvers

Operators often produce handles rather than final objects.

Examples include:

```text
create user
->
user identifier
```

```text
allocate memory
->
pointer
```

```text
book journey
->
ticket reference
```

```text
register organization
->
legal entity number
```

A handle is useful only when it remains:

```text
unique enough
in scope
persistent enough
resolvable
authorized
semantically stable
```

An identifier is not necessarily a locator.

A resolver is a specialized bridge operator:

\[
\rho :
H
\times
C
\to
L
\cup
\{\operatorname{failure}\}
\]

where:

```text
H
=
handle space
```

```text
C
=
resolution context
```

```text
L
=
locator or access descriptor
```

Resolution may fail through:

```text
ambiguity
collision
revocation
staleness
dangling reference
namespace mismatch
semantic drift
```

Reference continuity is therefore a first-class composition obligation.

## 18. Bridges and Representation Changes

A bridge operator converts between representational domains:

\[
b :
A
\to
B
\]

Examples include:

```text
parser
encoder
compiler
currency converter
sensor
actuator
protocol adapter
legal recognition process
chemical purification
transport terminal
```

A bridge contract should declare:

```text
what is preserved
what is transformed
what is discarded
what uncertainty is introduced
which references remain valid
which authority changes
```

Safe composition across domains depends on structure preservation, not merely successful conversion.

A parser that returns a syntactically valid value may still lose:

```text
units
provenance
precision
security labels
version information
```

The converted value may therefore be unusable by later operators even when its surface representation appears compatible.

## 19. Operator Refinement

One operator refines another when it can safely substitute for it under the declared contract.

Write:

\[
u'
\preceq
u
\]

when \(u'\) is a valid refinement of \(u\).

A standard refinement direction is:

```text
weaker or equal preconditions
stronger or equal success guarantees
stronger or equal frame guarantees
no additional externally relevant failures
no greater authority requirement
resource use within the declared bound
observations compatible with the original contract
```

Formally, a candidate refinement should satisfy obligations such as:

\[
P_u
\Rightarrow
P_{u'}
\]

\[
Q_{u'}
\Rightarrow
Q_u
\]

\[
\operatorname{Frame}_{u'}
\Rightarrow
\operatorname{Frame}_u
\]

A faster implementation is not necessarily a refinement if it introduces weaker reliability.

A cheaper implementation is not necessarily a refinement if it changes timing, precision, security, or failure behavior beyond the public contract.

Refinement gives a precise criterion for safe replacement.

## 20. Observational Equivalence and Operator Identity

Two realizations may differ internally while being equivalent relative to an observation regime.

Write:

\[
u
\equiv_{\pi,K}
v
\]

when \(u\) and \(v\) are indistinguishable under:

```text
π
=
permitted observation projection
```

```text
K
=
required guarantee set
```

The guarantee set may include:

```text
functional output
timing
cost
reliability
security
privacy
precision
failure semantics
resource bounds
```

Operator identity is therefore relative rather than absolute.

Two processors may implement the same instruction under an architectural observation regime.

Two payment providers may implement the same payment contract under one service-level regime but not another.

Two chemical procedures may be equivalent for product identity but not for:

```text
purity
yield
waste
safety
cost
scale
```

Therefore:

> Implementation replacement preserves operator identity only relative to declared observations and guarantees.

## 21. Context-Indexed Meaning

The same symbolic invocation may denote different transitions in different contexts.

\[
\llbracket a \rrbracket_c
\]

Context may include:

```text
namespace
protocol state
version
locale
privilege mode
active transaction
jurisdiction
input focus
current screen
organizational role
```

Versioning is therefore part of operator identity.

A robust identifier may include:

\[
(
\text{name},
\text{namespace},
\text{version},
\text{contract hash}
)
\]

This distinguishes:

```text
same label
```

from:

```text
same operational meaning
```

Semantic drift occurs when the identifier remains stable while the contract changes.

## 22. Mechanistic, Interpretive, and Strategic Operators

The common operator abstraction spans many domains, but not all realizations behave like machine instructions.

Three broad categories are useful.

### Mechanistic operators

Their semantics are externally specified and execution is largely governed by physical or computational mechanism.

Examples:

```text
machine instructions
cryptographic operations
database mutations
controlled physical procedures
```

### Interpretive operators

Authorized execution partly determines how the contract applies.

Examples:

```text
legal adjudication
medical assessment
regulatory review
professional certification
```

Their contracts may guarantee:

```text
procedure
evidence standards
decision authority
appeal rights
```

rather than one predetermined outcome.

### Strategic operators

Outcomes depend on interacting agents with independent objectives.

Examples:

```text
negotiation
auction
collective decision
market transaction
diplomatic process
```

Their semantics may require game-theoretic assumptions.

These categories share structured invocation and contract boundaries, but differ in how outcomes are generated and verified.

Human interpretation and strategic behavior should not be reduced to ordinary machine nondeterminism.

## 23. Operator Promotion

A repeated composition may be packaged as a higher-level operator.

Suppose:

\[
P =
u_n
\circ
\cdots
\circ
u_1
\]

A promoted operator \(U\) may satisfy:

\[
\llbracket U \rrbracket
\equiv_{\pi,K}
\llbracket P \rrbracket
\]

Promotion is justified when the path is:

```text
frequently repeated
semantically coherent
stable enough to contract
valuable to expose
costly to reconstruct
safely encapsulable
```

Promotion can reduce:

```text
search burden
invocation complexity
coordination cost
error rate
reference burden
```

But it can also conceal:

```text
intermediate choices
failure points
resource usage
governance decisions
```

Operator promotion is therefore an interface and governance decision, not merely an optimization.

## 24. The Principle of Least Transition

A safe system should expose the smallest transition family sufficient for the intended task.

This is the **principle of least transition**:

> Grant access to the narrowest operator contract that can accomplish the authorized objective.

This restricts:

```text
argument domains
authority scope
resource consumption
state effects
failure propagation
concurrency impact
```

Examples:

```text
transfer up to an approved amount
```

rather than:

```text
arbitrarily mutate account state
```

```text
write within an allocated region
```

rather than:

```text
write to any physical address
```

Least privilege restricts who may act.

Least transition also restricts what state changes their authority can denote.

## 25. Optionality as Future Executability

Let:

\[
\mathcal U_{\operatorname{exec}}(c)
\]

be the set of executable operator instances from configuration \(c\).

A resource buffer is valuable when it preserves a larger future set:

\[
|
\mathcal U_{\operatorname{exec}}(c_{\text{buffered}})
|
>
|
\mathcal U_{\operatorname{exec}}(c_{\text{depleted}})
|
\]

This gives a general account of optionality.

Examples include:

```text
cash reserves
free memory
schedule slack
spare transport capacity
legal continuity
institutional reputation
physical health
inventory buffers
```

Their value lies partly in preserving future operator availability.

A locally efficient transition may be globally harmful when it collapses the future executable set.

## 26. Operator-Constrained Reachability

Let:

\[
\mathcal U_t
\]

be the operator system available at time \(t\).

Let:

\[
c_0
\]

be the starting configuration.

Let:

\[
Y
\]

be a target condition.

The target is practically reachable when there exists a well-formed plan \(P\) such that:

\[
\Pr
\left[
\operatorname{Exec}(P,c_0)
\models
Y
\right]
\geq
\theta
\]

subject to:

\[
\operatorname{WellFormed}(P)
\]

\[
\operatorname{Typed}(P)
\]

\[
\operatorname{ContractCompatible}(P)
\]

\[
\operatorname{Resolvable}(P,t)
\]

\[
\operatorname{Authorized}(P,t)
\]

\[
\operatorname{ResourceFeasible}(P,t)
\]

\[
\operatorname{TemporallyFeasible}(P,t)
\]

\[
\operatorname{FailureSafe}(P)
\]

\[
\operatorname{InfrastructureAvailable}(P,t)
\]

where \(\theta\) is an acceptable reliability threshold.

Reachability is therefore a property of:

```text
starting configuration
target condition
available operator library
composition rules
reference system
authority system
resource system
infrastructure
time
acceptable risk
```

It is not a property of states alone.

### Bridge discovery

```text

```


### Operator promotion



## 28. Operator-System Evolution

Let:

\[
G_t
\]

be the practical transition structure at time \(t\).

Let:

\[
\Phi_t
\]

be the set of graph-modifying processes.

Then:

\[
G_{t+1}
=
\Phi_t
(
G_t,
\text{knowledge},
\text{investment},
\text{authority},
\text{coordination},
\text{maintenance}
)
\]

Operators can be:

```text
created
discovered
standardized
promoted
refined
reimplemented
versioned

deprecated

```

Operator systems are therefore historical and institutional objects.

Their persistence depends on:

```text

```

An uninterpretable or unmaintained operator may remain physically possible while ceasing to be practically available.

## 29. Measuring Operator-Set Quality

Counting operators is not enough.

A larger public operator set may increase:

```text
search complexity
ambiguity
overlap
security surface
documentation cost
versioning burden
selection error
composition uncertainty
```

Useful quality dimensions include:

### Coverage

How many valuable target conditions become reachable?

### Composability

How often do contracts connect without custom adaptation?

### Safety

How tightly are authority and effects bounded?

### Stability

How resistant are identities and contracts to semantic drift?

### Discoverability

Can agents find the appropriate operator?

### Verifiability

Can realizations be checked against contracts?

### Recoverability

Are failures observable and compensable?

### Maintenance cost

What infrastructure is required to preserve the operator?

A rough leverage metric is:

\[
\operatorname{Leverage}(u)
=
\frac{
\text{valuable reliable compositions enabled by }u
}{
\text{maintenance}
+
\text{cognitive cost}
+
\text{security cost}
}
\]

This is not a pure combinatorial count.

Invalid, redundant, dangerous, and uninterpretable compositions should not be treated as useful capability.

## 31. Verification

An operator is trustworthy only when its realization can be shown to satisfy its contract to an acceptable degree.

Possible verification methods include:

```text
formal proof
type checking
model checking
contract testing
simulation
measurement
audit
certification
quality control
reputation
appeal and review
```

Verification evidence is itself contextual.

A CPU instruction may be validated against an architectural test suite.

An API may be validated through contract tests and production observability.

A chemical procedure may be validated through analytical measurement.

An institutional procedure may be validated through records, review, and appeal.

The framework therefore distinguishes:

```text
contract truth
```

```text
evidence that a realization satisfies the contract
```

## 35. Design Principles

### Explicit invocation

Every operator should expose a decidable invocation language.

### Semantic typing

Types should encode meaning, not merely representation.

### Explicit frames

Contracts should define what may not change.

### Failure completeness

Every relevant failure should have declared state and recovery semantics.

### Least transition

Expose the narrowest sufficient transition family.

### Authority continuity

Composition should track capabilities as carefully as data.

### Reference continuity

Handles should preserve scope, resolution, and meaning.

### Observable identity

Replacement should be justified relative to declared guarantees.

### Contractual composability

Outputs, frames, resources, time, and failures should support safe chaining.

### Semantic maintenance

Documentation, versioning, and verification are part of operational infrastructure.

### Optionality preservation

Plans should account for the future executable operator set.

### Promotion with caution

Repeated paths should become primitives only when their abstraction boundary is stable and governable.

## 36. Core Propositions

### Proposition 1

An operator is not an arbitrary state relation. It is a constrained family of invocations with a stable contract and a maintained realization.

### Proposition 2

An operator specification, realization, availability, and executable instance are distinct statuses.

### Proposition 3

Syntax and representation types are insufficient for composition; semantic types and contract implications are required.

### Proposition 4

Frame conditions are essential because useful operators are defined partly by the state they are forbidden to modify.

### Proposition 5

Failure behavior is part of operator identity because it determines whether trajectories can terminate or recover safely.

### Proposition 6

Authority, resources, references, and time are components of execution configuration rather than external annotations.

### Proposition 7

Sequential composition is a proof obligation across types, postconditions, frames, resources, authority, references, time, failures, and interference.

### Proposition 8

Parallel composition requires compatible rely-guarantee conditions or another explicit interference model.

### Proposition 9

One operator refines another only when it safely preserves the original contract under the declared observation and guarantee regime.

### Proposition 10

Implementation replacement preserves operator identity only relative to explicit observations and guarantees.

### Proposition 11

Resolvers, parsers, gateways, sensors, adapters, and recognition procedures are themselves operators.

### Proposition 12

Information-producing operators transform knowledge state and must declare freshness, accuracy, and uncertainty.

### Proposition 13

Human and institutional operators may be interpretive or strategic rather than merely mechanistic.

### Proposition 14

A repeated path can be promoted into an operator when its boundary is stable, coherent, valuable, and maintainable.

### Proposition 15

Optionality is the preservation of a broad future executable operator set.

### Proposition 16

Practical reachability is a property of an operator system, execution context, target condition, and acceptable reliability threshold.

### Proposition 17

A rich operator theory does not replace theories of value, governance, invention, maintenance, or legitimacy.

## 37. Compact Definition

A practical operator is:

\[
u
=
(
\text{Access},
\text{Signature},
\text{Contract},
\text{Realization}
)
\]

It is **specified** when its signature and contract are defined.

It is **realized** when a maintained implementation satisfies that contract.

It is **available** when the realization is currently reachable.

A particular invocation is **executable** when:

```text
its syntax is valid
its arguments satisfy their roles and semantic types
its references resolve
its preconditions hold
its authority requirements are satisfied
its resource obligations are feasible
its time constraints hold
its realization is available
```

Two operator instances compose when:

```text
the first establishes the second's semantic inputs
the first preserves the second's assumptions
resources and authority remain sufficient
references remain valid
time windows remain feasible
failures have safe continuations
concurrent interference is controlled
```

A target is practically reachable when a valid plan of such instances achieves it with acceptable reliability.

## 38. Final Principle

An operator name is not yet an operator.

It must have a signature.

A signature is not yet a capability.

It must have a contract.

A contract is not yet an executable transition.

It must have a satisfying realization.

A realization is not yet available.

It must be pointable, reachable, authorized, interpretable, and maintained.

A parameter is not an arbitrary placeholder.

It occupies a role and must inhabit a semantic type.

A well-formed invocation is not necessarily executable.

Its preconditions, resources, authority, references, and time constraints must hold.

A successful local transition is not yet a safe trajectory.

Its outputs, frames, failures, and resource effects must compose with what follows.

A repeated trajectory is not yet infrastructure.

It must be stabilized, specified, verified, governed, and maintained as a reusable operator.

A finite agent cannot invoke arbitrary change.

It acts through a structured language of available transition contracts.
